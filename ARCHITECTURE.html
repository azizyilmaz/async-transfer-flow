Transfer System
Outbox + Kafka Architecture

This document explains the design of the money transfer system — why it was built this way, how data flows through it, and what each component is responsible for.

01 — The problem this system solves

Before — broken design

Database lock timeout

Every transfer request tried to UPDATE the same firm account row at the same time. The database locked that row for each update. When many requests arrived together, they all waited for the lock, and many of them timed out before they could proceed.

After — this design

Zero lock conflicts

The transfer API no longer touches the account row at all. It only writes a new row into the outbox_events table. A Kafka consumer then reads these events in order and updates the balance one at a time, so there is never more than one writer at once.

02 — How a transfer moves through the system

Step 1 — Firm sends a request

POST /api/v1/transfers

The firm's system calls the REST API with a firmAccountId, a customerAccountId, an amount, and a correlationId. The correlationId is a unique key chosen by the caller — it allows the system to recognise and ignore duplicate requests safely.

HTTP 202 Accepted idempotent
Step 2 — One database transaction writes two rows

transfers + outbox_events (PENDING)

Inside a single transaction, the service creates a record in the transfers table and a record in the outbox_events table. These two writes either both succeed or both fail together. The firm account row is never touched here. This is the core of the Transactional Outbox Pattern.

atomic no account lock
Step 3 — Outbox relay sends events to Kafka

PENDING → Kafka topic → PUBLISHED

A scheduled thread inside transfer-service runs every 500 ms. It reads all PENDING rows from the outbox table and publishes them to the transfer.initiated.v1 Kafka topic. The aggregateId (the transfer ID) is used as the Kafka partition key, so all events for the same transfer always go to the same partition in the same order. If publishing fails, it retries up to five times before marking the event as FAILED.

async at-least-once up to 5 retries
Step 4 — Balance Processor consumes events in order

Sequential consumption → balance update

The balance-processor service listens to the Kafka topic. Because firmAccountId is the partition key, all events for a given firm arrive in the same partition, in order. The consumer processes them one by one. Before updating the balance, it checks the processed_events table to make sure this transferId has not already been handled — this protects against Kafka delivering the same message twice.

sequential idempotent dedupe Dead Letter Topic
Step 5 — Firm checks the result

GET /api/v1/transfers/{id}/status

The firm can poll this endpoint at any time using the transferId from step 1. The status moves from INITIATED to PROCESSING to COMPLETED.

04 — System guarantees

No lock timeouts
Append-only writes

The transfer API only inserts new rows. It never updates the firm account row. Lock contention is gone.

Idempotency
correlationId

Sending the same request twice returns the same transferId and does not create a second transfer.

Exactly-once balance update
Deduplification

Kafka delivers at-least-once. The processed_events table ensures each transferId changes the balance only once.

No data loss
Transactional outbox

Transfer and outbox records are written in the same database transaction. If the app crashes after writing, the relay picks it up on restart.
